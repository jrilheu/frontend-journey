Clase 3 – Ramas, merge, conflictos y pull requests

Duración: 30–40 minutos

Objetivo de esta clase

Cerrar el modulo con el flujo colaborativo mas usado: crear ramas para trabajar sin romper main, integrar cambios con merge, entender conflictos basicos y usar pull requests como mecanismo de revision antes de fusionar.

Ramas: por que y como

Una rama es una linea de trabajo aislada. En vez de cambiar todo en main, creas una rama para una tarea puntual (por ejemplo feature/navbar).

git branch
git switch -c feature/navbar

Con eso creas y cambias a la rama en un paso. Para volver a la principal:

git switch main
Regla practica: una rama por objetivo pequeno (bug, mejora, refactor). Ramas cortas y con cambios concretos se revisan y fusionan mejor.

Flujo recomendado en una rama

  1. Partes de main actualizada: git switch main + git pull.
  2. Creas rama nueva: git switch -c feature/lo-que-sea.
  3. Trabajas y haces commits pequenos y claros.
  4. Subes la rama: git push -u origin feature/lo-que-sea.
  5. Abres Pull Request en GitHub hacia main.

Si necesitas traer cambios recientes de main a tu rama antes de terminar:

git switch main
git pull
git switch feature/lo-que-sea
git merge main

Merge basico

Cuando una rama esta lista, puedes fusionarla en main:

git switch main
git pull
git merge feature/navbar

Si no hubo conflictos, Git integra los commits y puedes subir:

git push

Luego puedes borrar la rama local (opcional):

git branch -d feature/navbar

Conflictos: que son y como resolverlos

Un conflicto aparece cuando Git no puede decidir automaticamente como combinar cambios sobre la misma zona de un archivo.

Flujo minimo para resolver:

  1. Intentas el merge (o pull) y Git marca archivos en conflicto.
  2. Abres el archivo y resuelves las marcas <<<<<<<, =======, >>>>>>>.
  3. Guardas el resultado final correcto.
  4. git add archivo-resuelto.
  5. git commit --trailer "Made-with: Cursor" para cerrar la resolucion del conflicto (si aplica).

Siempre revisa que la solucion compile o funcione; resolver conflicto no garantiza que la logica quede bien.

Pull Request: revision antes de fusionar

Una Pull Request (PR) compara tu rama contra main, muestra el diff y permite comentarios/revision antes de merge.

Checklist rapido para una buena PR

  • Titulo claro: que problema resuelve.
  • Descripcion corta: que cambiaste y por que.
  • Commits limpios y mensajes entendibles.
  • Pasos de prueba para quien revisa.

En equipos, la PR es tanto tecnica como comunicacional: reduce errores y alinea decisiones.

Consejo: evita trabajar varios dias en una rama enorme. Es mejor abrir PRs pequenas y frecuentes; se revisan rapido, causan menos conflictos y son mas faciles de revertir si algo sale mal.

Comprueba lo aprendido

Practica (20–25 min):

  1. Desde main, crea una rama feature/practica-rama.
  2. Haz 2 commits pequenos sobre un archivo de prueba.
  3. Sube la rama con git push -u origin feature/practica-rama y crea una PR en GitHub.
  4. Simula un conflicto (cambia la misma linea en main y en tu rama), resuelvelo y completa el merge.

Con esto cierras el modulo: ya tienes base local, remoto y colaboracion en ramas con PR.
