Clase 1 – Git básico: fundamentos y primeros commits (local)

Duración: 25–35 minutos

Objetivo de esta clase

Este módulo tiene 3 clases sobre Git y GitHub. En esta primera sesión trabajamos solo en tu máquina: entender qué es cada herramienta, tener Git instalado y configurado, crear un repositorio y registrar cambios con commits. Publicar en GitHub y colaborar con remoto lo verás en las clases siguientes.

¿Qué es Git y qué es GitHub?

Git es un sistema de control de versiones: guarda el historial de cambios de tus archivos y te permite volver atrás, comparar versiones y trabajar en ramas. Funciona en tu ordenador; no necesitas internet para hacer commits en local.

GitHub es un servicio en la nube (hay otros, como GitLab o Bitbucket) donde puedes alojar copias de tus repositorios Git, compartirlos y colaborar. Git es la herramienta; GitHub es un lugar donde usarla con más personas. En la Clase 2 conectarás tu repo local con un remoto en GitHub.

Regla mental: Git versiona carpetas y archivos de un proyecto. Cada “captura” coherente del estado que quieres guardar es un commit, con un mensaje que explique qué cambió.

Instalar Git y comprobar la versión

Si aún no lo tienes, instala Git desde la página oficial (git-scm.com/downloads) siguiendo el asistente de tu sistema (Windows, macOS o Linux).

Abre una terminal y ejecuta:

git --version

Deberías ver algo como git version 2.x.x. Si el comando no se reconoce, revisa que Git esté en el PATH o reinicia la terminal tras instalar.

Configuración inicial (una vez por máquina)

Git asocia cada commit con un nombre y un correo. Configúralos (usa el mismo correo que en GitHub si más adelante vas a subir código):

git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"

Para ver qué quedó guardado:

git config --global --list

Opcional: nombre de la rama por defecto

Muchos proyectos usan main como rama principal. Puedes fijarlo así:

git config --global init.defaultBranch main

Tu primer repositorio local

Crea una carpeta para practicar (por ejemplo mi-primer-repo), entra en ella y inicializa Git:

mkdir mi-primer-repo
cd mi-primer-repo
git init

Git creará una carpeta oculta .git con el historial del repositorio. A partir de aquí esa carpeta es un repo Git.

Tres zonas que debes conocer

  • Directorio de trabajo: tus archivos editables en la carpeta del proyecto.
  • Staging (índice): “escenario” donde eliges qué cambios formarán parte del próximo commit con git add.
  • Historial (commits): las versiones guardadas en el repositorio con git commit.

El flujo básico: status, add, commit

Crea un archivo de prueba, por ejemplo README.md, guarda contenido y ejecuta:

git status

Verás el archivo como untracked o modified. Para preparar el commit:

git add README.md

O, para añadir todos los cambios rastreados y nuevos en la carpeta actual (con cuidado en proyectos grandes):

git add .

Registra el snapshot con un mensaje claro:

git commit -m "Añade README inicial"

Consulta el historial:

git log --oneline

Cada línea es un commit con un hash corto y el mensaje. Repite el ciclo: editar → status → add → commit.

Archivos que no quieres versionar: .gitignore

Algunos archivos no deben entrar en Git: dependencias generadas (node_modules), secretos, cachés, binarios pesados. Crea un archivo llamado .gitignore en la raíz del repo y lista patrones (una por línea):

node_modules/
.env
*.log

Git dejará de ofrecerte esos archivos para el staging. En proyectos reales suele haber plantillas según el lenguaje o framework.

Consejo: escribe mensajes de commit en presente o imperativo corto (“Añade login”, “Corrige typo”), como si completaras la frase “Este commit…”. Evita mensajes vacíos como “cambios” o “fix”: en un mes no recordarás qué eran.

Comprueba lo aprendido

Práctica (15–20 min):

  1. Comprueba git --version y tu user.name / user.email con git config --global --list.
  2. Carpeta nueva → git init → archivo hola.txt → primer commit con mensaje descriptivo.
  3. Edita el archivo, vuelve a hacer status, add y un segundo commit. Revisa el historial con git log --oneline.
  4. Opcional: crea .gitignore e ignora un archivo temporal; verifica con git status que no aparece para añadir.
