Clase 1 – Accesibilidad web (a11y): fundamentos y HTML semántico

Duración: 25–35 minutos

Objetivo de esta clase

Este módulo tiene 2 clases sobre accesibilidad web (a11y) desde el frontend. En esta primera sesión sentamos bases: qué es la accesibilidad, cómo se relaciona con HTML semántico, estructura de encabezados, texto de enlaces, imágenes con alt, contraste y navegación por teclado. La Clase 2 profundiza en ARIA cuando hace falta, formularios, patrones útiles y cómo probar.

Qué es la accesibilidad web

La accesibilidad web es el conjunto de prácticas que permiten que más personas usen tu interfaz: personas con discapacidad visual, auditiva, motriz o cognitiva, pero también quien usa solo teclado, pantallas pequeñas, zoom alto o conexiones lentas.

No es un “extra” de diseño: es calidad y alcance. En muchos países la accesibilidad digital es requisito legal para sitios públicos; en productos privados mejora SEO, mantenimiento y satisfacción de usuario.

Regla mental: si el HTML comunica bien la estructura y el contenido, ya resolviste gran parte del camino. ARIA y CSS refinan, pero no sustituyen un documento semántico coherente.

POUR y WCAG (visión rápida)

El modelo POUR resume cuatro principios: contenido Percepcible, Operable, Understandible (comprensible) y Robusto.

Las WCAG (Web Content Accessibility Guidelines) son pautas técnicas; lo habitual en proyectos es apuntar a nivel AA como referencia. No hace falta memorizar criterios al inicio: si practicas semántica, teclado, contraste y textos alternativos, ya caminas en la dirección correcta.

HTML semántico y landmarks

Usa elementos con significado: header, nav, main, footer, section, article, listas reales (ul/ol), button para acciones y a para navegación.

Debe existir un solo <main> por página (salvo casos muy específicos). Las tecnologías de asistencia usan estos landmarks para saltar bloques.

Jerarquía de títulos (h1–h6)

Los encabezados crean un índice del documento. No saltes niveles sin motivo (por ejemplo de h1 a h4) y no uses títulos solo por tamaño de fuente: el tamaño se ajusta con CSS.

Una sola h1 por vista es una buena práctica habitual para reflejar el tema principal de la página.

Enlaces e imágenes

El texto del enlace debe decir a dónde va o qué hace. Evita “clic aquí” o “leer más” sin contexto.

En imágenes con significado, alt describe la intención; si son puramente decorativas, usa alt="" para que el lector de pantalla las ignore.

Teclado y foco visible

Mucha gente navega sin ratón. Comprueba que todos los controles interactivos reciban foco con Tab y se activen con Enter o Espacio según corresponda.

No elimines el :focus del navegador sin sustituirlo por un indicador visible equivalente. Ocultar foco rompe la orientación de quien usa teclado o lectores.

Contraste y zoom

El texto debe leerse con suficiente contraste frente al fondo (herramientas del inspector o extensiones ayudan a medirlo).

Prueba tu layout con zoom al 200 %: el contenido debería seguir usable sin solaparse de forma que impida leer o pulsar.

Consejo: reserva una sesión corta solo con teclado (sin ratón) en tu página. Lo que falle ahí suele ser el primer bug de accesibilidad que notan los usuarios reales.

Comprueba lo aprendido

Práctica (15–20 min):

  1. Marca una página con header, nav, main y footer si aún no lo están.
  2. Revisa la jerarquía de h1–h3 y corrige saltos o títulos usados solo por estilo.
  3. Mejora el texto de dos enlaces genéricos y revisa los alt de tres imágenes.
  4. Navega la página solo con teclado y anota un problema de foco o orden.

En la Clase 2 verás ARIA con mesura, formularios accesibles, patrones como “saltar al contenido” y cómo probar con herramientas.
