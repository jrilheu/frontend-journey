Clase 2 – Accesibilidad web (a11y): ARIA, formularios y pruebas

Duración: 30–40 minutos

Objetivo de esta clase

En la Clase 1 viste bases de semántica, teclado y percepción. Aquí completas el módulo con ARIA (solo cuando el HTML no basta), formularios bien etiquetados, patrones útiles y una rutina de pruebas sencilla que puedes repetir en cada entrega.

ARIA: primera regla

ARIA (Accessible Rich Internet Applications) añade información accesible cuando los elementos nativos no alcanzan. La primera regla de ARIA es: si existe un elemento HTML nativo adecuado, úsalo en lugar de reinventar roles con div y ARIA.

Ejemplos habituales donde ARIA ayuda: widgets complejos (acordeones, pestañas), regiones dinámicas con aria-live, o anunciar estados con aria-expanded en controles desplegables.

Formularios accesibles

Cada control debe tener etiqueta visible asociada con <label for="id"> o el control envuelto por el label. Los errores deben ser perceptibles (texto, no solo color) y, si puedes, enlazarlos al campo con aria-describedby.

<label for="email">Correo</label>
<input id="email" type="email" autocomplete="email" />

Evita placeholders como única etiqueta: suelen tener bajo contraste y desaparecen al escribir.

Saltar al contenido (skip link)

Un enlace al inicio que salta al main ahorra tabulaciones repetidas en cada página para usuarios de teclado y lectores de pantalla.

<a class="skip-link" href="#contenido">Ir al contenido</a>
...
<main id="contenido" tabindex="-1"> ... </main>

El enlace puede estar oculto visualmente hasta recibir foco; lo importante es que sea focalizable y visible al activarse.

Cambios dinámicos y movimiento

Si una zona se actualiza sin recargar la página (por ejemplo resultados de búsqueda), un contenedor con aria-live="polite" puede anunciar cambios sin interrumpir de forma agresiva.

Respeta @media (prefers-reduced-motion: reduce) para reducir animaciones largas o parallax; muchas personas lo tienen activado en el sistema.

Cómo probar (sin volverse experto overnight)

  • Teclado: Tab/Shift+Tab por toda la página; sin trampas de foco.
  • Lighthouse (Chrome): pista rápida de accesibilidad; no sustituye revisión manual.
  • Lector de pantalla (NVDA en Windows, VoiceOver en macOS/iOS): 10 minutos por vista para detectar orden de lectura absurdo.
  • Contraste y zoom como en la Clase 1.
Consejo: anota en tu checklist tres comprobaciones fijas antes de cada release: teclado completo, foco visible, etiquetas en formularios. Eso solo ya evita una parte grande de incidencias frecuentes.

Comprueba lo aprendido

Práctica (20–25 min):

  1. Corrige un formulario: cada campo con label asociado y mensaje de error legible (no solo color).
  2. Añade un skip link funcional hacia tu main.
  3. Simula una actualización de estado (por ejemplo “Guardado”) con una región aria-live="polite".
  4. Pasa Lighthouse accesibilidad y corrige al menos un aviso concreto.

Con las dos clases tienes una base sólida para seguir profundizando en patrones complejos y auditorías WCAG cuando lo necesites.

Ruta del módulo (2 clases)

  • Clase 1: conceptos POUR/WCAG; HTML semántico; encabezados; enlaces e imágenes; teclado y foco; contraste y zoom.
  • Clase 2: ARIA cuando hace falta; formularios (label, errores); patrones (skip link, aria-live); movimiento reducido; pruebas básicas.
