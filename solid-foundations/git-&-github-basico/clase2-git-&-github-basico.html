Clase 2 – GitHub remoto: conectar, subir y traer cambios

Duración: 25–35 minutos

Objetivo de esta clase

En la Clase 1 trabajaste Git en local. Ahora vas a conectar ese flujo con GitHub: crear un repositorio remoto, enlazarlo desde tu repo local, subir cambios con push y traer cambios con pull. Al terminar, ya tendrás un flujo minimo de sincronizacion.

Antes de empezar

Necesitas una cuenta de GitHub y Git instalado. Comprueba en terminal:

git --version

Opcional pero recomendado: autenticarte con GitHub CLI (gh auth login) o tener listo tu metodo HTTPS con token (o SSH) para evitar problemas al hacer push.

Repositorio remoto en GitHub

Crea un repositorio nuevo en GitHub (vacio, sin README si ya tienes proyecto local). Copia la URL remota que te da GitHub, por ejemplo:

https://github.com/tu-usuario/mi-primer-repo.git

Desde tu proyecto local, enlaza ese remoto con el nombre origin:

git remote add origin https://github.com/tu-usuario/mi-primer-repo.git

Verifica que quedo bien:

git remote -v
Regla mental: origin es solo un alias (nombre corto) para una URL remota. Puedes tener mas de un remoto, pero el primero casi siempre se llama origin.

Primer push y rama de seguimiento

Si ya tienes commits locales, subelos por primera vez:

git push -u origin main

El flag -u deja configurada la rama local para seguir a origin/main. Desde ese momento, normalmente te bastara con usar:

git push
git pull

Si tu rama principal se llama master, reemplaza main por master en el comando.

Cuando usar clone

Usa git clone cuando el repositorio ya existe en GitHub y quieres bajarlo completo a tu maquina:

git clone https://github.com/tu-usuario/mi-proyecto.git
cd mi-proyecto

clone ya configura el remoto origin automaticamente. En ese caso no necesitas ejecutar git remote add origin ....

Ciclo diario basico con remoto

  1. git pull para traer cambios remotos recientes.
  2. Editas archivos en local.
  3. git status para revisar el estado.
  4. git add . (o archivos concretos) para staging.
  5. git commit --trailer "Made-with: Cursor" -m "mensaje claro".
  6. git push para subir al remoto.

Este ciclo evita divergencias grandes y te ayuda a integrar cambios poco a poco.

Errores comunes y como reaccionar

  • non-fast-forward al hacer push: alguien subio cambios antes. Haz git pull --rebase (o git pull segun flujo), resuelve conflictos si aparecen y vuelve a git push.
  • remote origin already exists: ya habias creado ese remoto. Revisa con git remote -v; si hace falta cambiar URL usa git remote set-url origin NUEVA_URL.
  • auth failed: revisa credenciales/token/SSH y permisos del repo.
Consejo: antes de empezar una sesion de trabajo, haz git pull. Antes de cerrar, haz git push. Mantener ciclos cortos reduce conflictos y sorpresas.

Comprueba lo aprendido

Practica (15–20 min):

  1. Crea un repo en GitHub y enlazalo con un repo local usando git remote add origin ....
  2. Sube tu rama principal con git push -u origin main.
  3. Haz un cambio pequeńo, crea commit y subelo con git push.
  4. Haz un cambio desde GitHub (por ejemplo editar README en web) y traelo con git pull.

En la Clase 3 daras el siguiente paso: ramas, merge, conflictos y flujo con pull request.
